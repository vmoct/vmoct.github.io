Electro-Caddy-Uno (or Electro-Cadduino) – Electronic Golf Caddy

 

  

 

An Arduino based smart electronic caddy for carrying a golf bag.

This remote controlled robotic upgrade to an existing electronic golf caddy chassis with two DC motors has proportional-integral (PI) feedback speed control, sonar object (collision) detection within a range of less than 5 yards (now decommissioned) and a large (6 inch) 2 digit 7 segment LED display (Cart I) or a 2 screen 128x64 pixel LCD panel (Cart II) for viewing cart operations from a distance.

As well as measuring cart speed, it records distance between stops or distance between points, total distance per round, instantaneous current and voltage (therefore wattage) and total wattage per round. It can be set to run for various yardages before auto stopping.

Target speed (in AUTO or CRUISECONTROL control mode) or power output to motors (in MANUAL or DIRECTDRIVE control mode) are each adjustable by the increment/decrement buttons. A small 16x2 character LCD screen on the user push handle shows cart operations, including target and measured speed, output percentage to the motors and various measured or calculated values. The large LED/LCD display shows (momentarily) cart state and some measured values depending on the cart state, including yard measurements.

The user input interface is minimalist with two switches (POWER ON/OFF and DIRECTDRIVE/CRUISECONTROL select) and a 8 button (START/STOP, INC, DEC, LEFT, RIGHT, 40 Yards Start/Countdown, 60 Yards Start/Countdown and 20 Yards Countdown/MODE SET) remote control panel (the first 3 buttons of which are duplicated at the local control panel on the cart push handle, which also includes a large STOP/START button for easy thumb control). All user input events are displayed, along with changed data values, momentarily, on the big LEDs or LCDs.

The cart can be set to automatically stop after 20 yards initially (MODE SET button while the cart is in the RUN state) but this value can be incrementally adjusted upwards any time before the automatic stoppage with added values of 10 yards for every button press. This same strategy applies to the 40 Yards Start (20 yards increment) and 60 Yards Start (30 yards increment) buttons except that in the cart STOP state, these buttons will also start the cart towards the countdown yardage. [In the STOP state, the MODE SET button puts the cart into the reverse direction for additional features – see below.]  The countdown yards-to-go is displayed on the big LEDs or LCDs and can be seen at a good distance.

The cart has a motor soft start and a motor soft stop (timed) but the collision-detect (within 2 yards) forces a stop more abruptly. Collision-detect only works in CRUISECONTROL mode (use DIRECTDRIVE mode if following another person closely on a cart trail) but a LED (modified flashlight) display will always show when objects are within 4 or 3 or 2 yards boundaries (3 different rows of LEDs with proportional frequency flashing rates). The cart will turn left or right while in the RUN state.

In STOP state, the mode button will set the motor direction to reverse for backing out of a collision-detect state or backing out from being ahead of someone else’s golf ball. The mode button toggles the motor direction when in the STOP state and when going from reverse to forward direction some cart values (targets, for example) are stored in EEPROM for loading at power up. In this special reverse mode state, the values used to increment/decrement the wheel power outputs while turning left or right can be edited to allow for more or less aggressive turns.

Control states of the Cart and User control of the caddy

                 

The above pictures show the remote and local control panels. The local pushbuttons STOP/START, DEC and INC are duplicated on the remote as buttons [2],[3] and [4] respectively. The DD/CC control mode SPST switch is only available on the local panel. The following is a description of the actions taken by the CARTSYS software for these user inputs.

The control system has six states – RUN, STOP, LEFT_TURN, RIGHT_TURN, SOFT_START & SOFT_STOP, where the last four are transitory states which are active only for a few seconds. The control system has two control modes – DD (direct drive) or manual control of motor output by percentage, and CC (cruise control) or feedback control of the measured rolling speed in inches / second. This control mode can be changed at any time, either in STOP state or in the RUN state, in which case the controller software uses motor output tracking to provide for bump-less transfers between the control modes. The large display will show either load AMPS and speed IN/SEC or YARDS – see toggling this mode with remote button [5].

The cart can also be placed in reverse direction (using remote button [1]) where all of the forward direction operations are available, except increment and decrement of targets. In reverse direction and STOP state, the left turn and right turn motor output increments/decrements can be edited (using the INC/DEC buttons with the edited values shown on small LCD screen) to provide different wheel turn aggression ratios. Upon exiting reverse direction, (using remote button [1]), these turn adjustment values, plus the DD and CC targets, are saved to EEPROM. These values are loaded from EEPROM at power up.

The control software keeps track of two accumulated yardage values, total for the round (between power ups) and a short accumulated yardage value. This second value has a reset mode, either reset after every stop, or never reset, which allows for measuring yardage between points on the golf course (actually up to 999 yards, as it is reset then to properly display on the small LCD screen). Note that this reset mode can be toggled by remote button [6] in the STOP state.

General Hardware Description

The system has 3 Arduino microcontrollers, one Uno for the cart processing (hereafter known as CARTSYS) and one Nano each for the big LEDs or LCDs (now known as BIGLED) and the collision detection system (now known as SONAR). Communication between the systems is simple. Two bits from SONAR are sent to CARTSYS which indicate a distance zone for objects heard (Zone 3 – out of range, Zone 2 – object between 3 and 4 yards away, Zone 1 – object between 2 and 3 yards away and Zone 0 – object less than 2 yards away (from the front wheel of the cart). Serial communication is used from CARTSYS to BIGLED (one simplex Tx to Rx signal). One or 2 bytes of binary data are sent to generate a message on BIGLED – 1 byte for event type only (Ex. ‘SP’ for STOP state) or 2 bytes for event plus data (Ex. ‘O^’ plus ‘65’ which means motor output incremented to a new value of 65 %). Automatic time delays are added after updated values by BIGLED for proper viewing of sequential values (text or numeric).

CARTSYS contains 2 stackable pulse width modulation (PWM) 10A DC motor driver boards (Cyton MD10). The PWM signals are generated by the Uno’s hardware timing system as directed by the control software (within a value range of 0 to 255 which is computed from percentages that are used by the user interface). CARTSYS contains a hall-effect total load current sensor (+-20A) and battery voltage (typically 14-15V with lithium batteries) sensing through a simple voltage divider to the 0-5V A/D converter input of the Uno. The wireless remote panel (a small hand-held unit easily clipped to a pants belt loop) is a simple RF AM (8 bits 433 Mhz) device with corresponding receiver which decodes one of eight button presses. Wireless range is not large, but is useful up to at least 50-60 yards. The remote transmitters/receiver pairs are device data synced so that they do not interfere with each other.

The speed/distance sensor comprises a circular metal cutting wheel disc with nine small round holes at evenly spaced circular positions. The backside of the cutting wheel disc has been taped with a reflective silver tape (heating duct tape). The disc is mounted to the driving cog or sprocket on the right hand wheel of the cart. A combined infrared LED transmitter and phototransistor receiver semi-conductor device (OPB702) has been mounted to transmit and received infrared light waves directly at the position where each of the nine holes of the rotating disc pass the sensor. A standard (555 timer) one-shot circuit conditions the receiver pulse to avoid jitter or bounce as the light passes over the round hole. It provides a one-time sharp pulse edge to a hardware interrupt pin (PIN 2) on the Arduino board per hole of the disc. The length of the one-shot time period has been adjusted (to approximately 30 msec) so that no re-trigger on the same hole will happen at low walking speeds and the one-shot period is short enough so that each hole will be triggered at higher walking speeds. This involved calculations based on the physical size of the driving wheels and the hole placements on the disc. The output of the photo-transistor contains an LED for visual diagnostics but the circuit board must be exposed from under its plastic covering (near the top front of the right strut of the cart) to see it flash. The hardware interrupt pin used (UNO pin 2 – INT0) has the highest priority in CARTSYS as it will interrupt the system timer interrupt, which is used for all precise or semi-precise timing functions. [Note – Cart II was built with a 8 hole tachometer disk and a separate infrared LED transmitter on one side of the slotted disk and an infrared photo transistor on the other side of the disc. This is a much easier and mechanically simpler arrangement, and the 555 timer sensing circuit is slightly different. ]

SONAR’s Nano processor (same features as a Uno except in a small form factor) handles the sonic transmitter/reflector device. Periodically it triggers a pulse out of the transmitter and waits for a response pulse edge (which will return the longest pulse if no object reflects it back within about 5 yards). The longer the time between trigger edge and response edge, the further the object is away and of course, the closer the object the sooner the response edge is received. Again a hardware interrupt is generated (on pin 2) at receipt signal edge and this is timed relative to the trigger edge by the system timer interrupt system (the same mechanism that is used for timed measurements and control in CARTSYS). The distance between the cart and any close objects (within 4 yards) is calculated based on the sonic time travel and the 2 bit (4 value) zone signal is sent to CARTSYS at various rates depending on the distance zone. Attached to another cable to SONAR is a modified LED flashlight case which has four rows of flashing LEDs for zones 0 to 2, where zone 0 (collision detect) cause the bottom two rows of the flashlight to flash while only one row is flashed for zones 1 and 2. There is a small (homemade) transistor driver board in the flashlight driving each of the LED rows. This flashlight LED system is strictly for diagnostic and visual monitoring of the sonic system which has proven to be perhaps temperamental in the real world (golf course) environment.

BIGLED contains a Nano processor board which is connected through the I2C interface to two chainable 7 segment LED driver boards. These driver boards contain serialized latches to hold the 8 bit data for each digit and 8 transistor drivers to light up the individual LEDs. There are built in resistors in these driver boards to limit the current through each LED but they are of a low value because the board has been designed to work with regulated 12V and 5V supplies. Since the lithium battery used in the cart nominally puts out at least 15V, resistors have been added in series with each of these LEDs to limit the current (and brightness). Mechanically this was convenient as the driver boards are mounted on the back side of the 6 inch LED digits and 4 signals go upwards to the big digit pin connectors and four go downwards to the lower pin connectors. However the order of the connections between these two boards does not correspond properly segment to segment in this mechanical situation. The software re-maps the incoming segment data to the correct physical pin on the big LED digits. There are no interrupts directly used in BIGLED software code (other than the hidden timer and serial communication interrupts). After initialization the code simply scans the serial communications Rx stream for incoming commands and data and displays the values with or without post-display delays which are customized for each type of display.

As a working prototype, virtually all connections and electronic boards in the cart have been ruggedized using soldered connections and hot-melt glue or screw connectors. Cables used for communication between the unit boxes all have male/female connectors for easy disconnection for re-programming, diagnostics or just disabling. CARTSYS will run independently without BIGLED or SONAR attached. The small LCD screen gives all the information required to monitor and change the system control actions.

More Detailed Electronic Description

The Arduino Uno has 20 input/output pins, 18 of which are used by the cart control system.  Several are dedicated by unchangeable hardware specifics, such as D11 (PWM output to left motor), D3 (PWM output to right motor), D2 (tachometer input – requires an interrupt), D1 – UART TX to LED/LCD driver boards (Arduino Nano) and A4/A5 – I2C bus SDA/SCL digital outputs.

A0 (total cart load current sense) and A1 (battery voltage sense) are analog inputs while A2 (local panel DD/CC switch) is a digital input. D13 (digital output) is used as the motor driver direction bit.

The rest (D4 – D10 and D12) are used as the various control buttons from the local or remote panels. The first three (D4 –D6) are hard wired in the OR sense between the local panel (INC,DEC,S/S) and buttons (4,3 and 2 of the remote) to save input pins. D7-D10 and D12 receive remote buttons 1,5,6,7 and 8 respectively.

The gate on the RF relay board that drives each relay coil low is used to drive the 74HC14 Schmitt Trigger gate which passes it along to the corresponding UNO digital input. Note that the relay on the RF receiver board is not used (as it would cause switch de-bouncing problems) and it could not be removed physically. It will click when a remote button is pressed, but it is not used by the hardware here. The voltage divider resistors reduce the 12V relay coil voltage to a little less than one-third of the value (the cart Lithium battery puts out as much as 16V) so that the 5V 74HC14 is safe from damage. The local panel input gates (again 74HC14) have typical old-style switch de-bouncing through a resistive/capacitive network. This reduces the software complexity and uncertainty about these input switch hits. As mentioned previously, three of these inputs are hardwired ORed by the 74HC32 gate to save digital inputs at the UNO. Also shown is the input for the static SPST DD/CC control mode switch which does not need de-bouncing.

The next schematic shows in more detail the actual gate wiring for the Cart II interface between the Uno and the local/remote panels. This board (in the main control box) uses the Uno’s 5V regulator and it only connects to the switch wiring in the cart handle – there is no logic in the cart handle, avoiding transmission line effects between gates.

This board has its own local 5V regulator (LM340) since it also drives the switch de-bouncing board located in the handle of the cart (unlike cart II where the switch de-bouncing board is within the main control cabinet) . A voltage divider (1:4) is used for the voltage sense (0-20V down to 0-5V) on A1. A0 receives the voltage from the Hall Effect current sensor (0 to +/-20V) which measures the total current used by the cart control system. The 10 bit Uno A/D converter will get values only in the range of 512 to 1023 (for 0 to +20A) since negative currents will never occur.

The next schematics show the tachometer disk signal conditioning circuit (Cart I and then Cart II). The purpose is to provide a stable sharp rising edge to the interrupt pin 2 of the Uno. The pulse must last long enough to cover the time that the infrared led/phototransistor circuit engages the passing hole/slot at slow walking speeds but not long enough to prevent the next hole/slot from being sensed at the highest walking speeds. The following standard 555 time circuit is perfect for this task. The pulse time is set to about 30 msec. The IR Detected LED is used for visual confirmation of pulses.

LED and LCD Electronic Description

The LED and LCD displays show the same information (apart from no decimal point shown on the LCD) in 7 segment digit format, which allows for some text characters besides the digits 0-9. The LED displays have large 6 inch digits while the LCD displays have a 128x64 pixel display where the 7 segment digits are drawn graphically.

The LED displays are driven electronically by the SparkFun Large Digit Driver, details of which can be found at: https://www.sparkfun.com/products/13279. This site has excellent documentation on how to use this driver, which has been connected to an Arduino Nano processor board for use here. The site also has a code samples for driving the displays.

There are three dedicated digital output lines (Clock, Latch & Data) used by the software to drive these two displays, which are connected in serial fashion to three signals (Nano D2, D3 & D4). The circuit below (from SparkFun) shows the hardware details where some modifications have been added.

First, 330 ohm resistors have been added in series with the 15 ohm current limiting resistors to each LED segment, except the decimal point (DP), which required more resistance. This is because the cart battery voltage is nominally 15V whereas the board is designed for a regulated 12V as is. Vf, or diode string forward voltage, was measured to be around 11V (but much less for the DP).

Second, the small driver logic boards and these resistors were mounted easily on the back of the large digits in such a way that they provided the ‘wiring’ as is. As a result, JP header signals were mismatched as to actual segments A to G. This mismatch was re-mapped in software. Generally speaking about 10 ma is provided to each LED segment when turned on.

The LCD displays are ST7920 128x64 pixel graphic displays with a light background and dark foreground for good viewing in sunlight. They both are connected by the SPI bus. Unfortunately, due to an issue in the ST7920 hardware signal interfacing, they both cannot be connected to the Nano SPI hardware bus at the same time without cross interference (adding two external AND gates can solve the problem). Therefore, one is driven by the hardware SPI bus of the Nano and the other is driven by software emulation of the SPI bus (a little slower but fast enough for this application).

The following diagram shows a proper LCD signal interface with a shared SPI bus. This image is from

https://www.eevblog.com/forum/projects/how-to-properly-connect-lcd-128x64-via-spi/

 

In the case of the cart Nano LCD driver, the hardware SPI connects Nano pin 11 MOSI to R/W of the LCD, pin 13 SCK to E of the LCD, pin 10 to RS of the LCD and pin 12 to the RST of the LCD.

The software emulated SPI connects Nano 8 pin to E of the LCD, pin 7 to R/W of the LCD, pin 6 to RS of the LCD and pin 5 to RST of the LCD.

The 7 segments are drawn graphically using the excellent ‘olikraus u8g2’ software library, which has support for a large number of displays using either a hardware SPI bus or a software emulated SPI bus. Each segment drawn is a simple rounded box carefully placed, and sometimes dynamically enlarged or shrunk, to provide the largest digits possible.
